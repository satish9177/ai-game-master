## Context

ADR-0047 through ADR-0050 built a complete generated-objective pipeline for the
**prompt-generated first room only**:

- `assembleObjective` is the sole safety/satisfiability gate (parse → schema →
  satisfiability → text sanitization → build `QuestSpec`).
- `buildGeneratedObjectiveAttachment` wraps the generator call and the assembler
  in a total `try/catch` that always returns `null` on any failure.
- `enrichObjectiveTarget` (ADR-0048) promotes exactly one eligible object in the
  assembled room so the generator has an eligible candidate; it was purposefully
  limited to the prompt-generated first room (`buildPromptGeneratedRoomSource` sets
  `enrichObjectiveTarget: true`; adjacent sources do not).
- `canAttemptOptional` (ADR-0050) gates optional provider calls against the shared
  room-generation counter without incrementing it.
- `selectObjectiveGenerator` selects the real or fake provider at startup.

When the player walks through a door into a freshly generated adjacent room, that
room receives no local objective today. The quest tracker goes blank; NPC hints revert
to the generic greeting. The world feels purposeless beyond the first room.

**The gap:** adjacent generated rooms — generated by `AdjacentRoomPregenerator` using
a `FakeRoomGenerator` (or, in the future, a real provider) — are never enriched for
objective eligibility, and the App never calls the objective pipeline at room entry.

**Why not fix this during warming?** `warmAdjacent` is called fire-and-forget by the
pregenerator after room entry and after navigation. Its design invariant is
**provider-free and network-free**: the generated-play adjacent factory uses
`FakeRoomGenerator` precisely so background warming never makes real API calls. Putting
objective generation in the pregenerator or `GeneratedRoomSource` would violate this
invariant, couple the generation layer to the `ObjectiveGenerator` port (a boundary
violation), and create hidden background provider spend the usage guard does not cover.

**Why not fix this in `NavigationService`?** Navigation must stay a pure
resolve-before-append mover. Injecting objective generation couples it to the
`ObjectiveGenerator` port and the usage guard, breaking its single responsibility.

**Solution:** attach objectives **in the App composition root on player room entry**
(in `handleNavigate`), gated by the same `canAttemptOptional` / `provenance` rules
already used for room #1. This is a wiring change only; every safety and domain layer
is unchanged.

ADR-0050 explicitly pre-authorized this: *"When `generated-objective-per-room-v0` adds
objective generation at each room navigation, the same function [`canAttemptOptional`]
can be called in the adjacent room resolution path with the then-current
`usageCountRef.current`. No domain change is needed; the wiring site changes, not the
guard model."*

## Decision

### Core invariant

**Background adjacent warming is and remains provider-free.** The `AdjacentRoomPregenerator`
and `GeneratedRoomSource` are never given the `ObjectiveGenerator` port. No objective
generation call is ever made during `warmAdjacent`. This invariant must not be weakened
in this slice or in any future slice without an explicit separate ADR.

### Provenance propagation

`AdjacentRoomPregenerator.resolveRoom` currently produces a `ResolveRoomResult` that
carries the resolved room but discards provenance after caching. The App currently has
no way to know whether a navigated room was `generated`, `repaired`, or `fallback`
without re-running assembly.

**Add `provenance?` to `ResolveRoomResult.ok` and `NavigationResult.navigated`.** The
pregenerator records provenance per `roomId` in an internal map and exposes it on cache
hits. `NavigationService` passes it through to `NavigationResult` without interpreting
it. The App reads it at the navigation call-site.

Provenance for authored/registry rooms is `undefined`; these rooms are never eligible
for objective attachment regardless.

### Objective-target enrichment for generated-play adjacents

The prompt-generated first room sets `enrichObjectiveTarget: true` in
`buildPromptGeneratedRoomSource`. Adjacent rooms constructed by the example/authored
pregenerator must never receive this flag (authored world safety).

**Enable `enrichObjectiveTarget: true` on the `GeneratedRoomSource` factory passed
to the generated-play `AdjacentRoomPregenerator` only** — the lambda captured inside
`handlePrompt`. The authored-world pregenerator at module scope is untouched.

### Eligibility policy

A room is eligible for per-room objective attachment if and only if all of:

1. The current `ActivePlay` has `objectivesPerRoom === true` (set only on the
   prompt-generated play; never on authored bootstrap or restored plays).
2. `result.provenance === 'generated'` (clean generated room; repaired/fallback rooms
   and authored/registry rooms are not eligible).
3. The room has not already been attempted this session: `roomId` is not in the
   per-room objective memo (see below).

### On-enter attachment (async, non-blocking, guarded)

Room transition must not wait for objective generation. The flow in `handleNavigate`:

```
result = await navigation.navigate(...)
if result.status === 'navigated':
  setActivePlay(...)     ← room is visible immediately; no door lag
  setRoomEntrySeq(...)
  warmAdjacent(result.room)
  refreshDerivedViews(result.state)

  // Async attach — fire-and-forget within a guard
  if eligible(result):
    capturedRoomId = result.room.id
    capturedSessionId = activePlay.sessionId
    void (async () => {
      if canAttemptOptional(count, config):
        attachment = await buildGeneratedObjectiveAttachment(room, objectiveGenerator)
      else: attachment = null (budget exhausted)

      memoize perRoomObjectiveMemo[capturedRoomId] = attachment

      // Discard if player moved away before resolution
      if currentActivePlay.room.id !== capturedRoomId ||
         currentActivePlay.sessionId !== capturedSessionId:
        return   // stale result; do not apply

      questSpecRef.current = attachment?.questSpec ?? null
      setQuestHints(attachment ? { hint, completionHint } : null)
      refreshDerivedViews(currentState)
    })()
```

A stale objective result (player moved before the `~12 s` timeout) is silently
discarded. It is never applied to the wrong room.

### Per-room objective memo

A `Map<string, GeneratedObjectiveQuestAttachment | null>` held in a ref on the App,
keyed by `roomId`, scoped to the current play session.

- **`null` is cached.** A room that yielded no objective (budget skip, provider failure,
  `assembleObjective` drop, or generator null) is recorded as `null`. Revisiting never
  retries the provider; the tracker simply stays hidden.
- **Room #1 is seeded** into the memo from the `handlePrompt` result so re-entering
  room #1 restores its tracker from the memo without a new provider call.
- **New play starts fresh.** The memo is reset (or a new ref is created) when
  `handlePrompt` begins a new generated play.
- **Sync restore on revisit.** If `roomId` is already in the memo, apply the cached
  attachment synchronously on room entry — no async, no provider call.

### QuestTracker / NPC hint behavior

- `questSpecRef` and `questHints` are swapped on every room entry to reflect that
  room's local objective only. On entering a room with no objective (or while the
  async call is pending) they are set to `null` — the tracker is hidden and the NPC
  greeting is the default.
- `computeDerivedViews(state, questSpecRef.current, journalSpec)` is unchanged; it
  already projects whatever spec is current.
- The authored demo QuestSpec (`demoQuestSpec`) is carried on `activePlay.questSpec`
  for the example play, consulted by `questSpecRef` on each navigation, and is not
  affected by this feature (`objectivesPerRoom` is `false` for that play).
- **`QuestTracker` shows only the current room's local objective.** There is no
  multi-objective panel in v0; the tracker is hidden between the room-entry reset
  and the async attach completing.

### Usage-guard policy (no change to domain model)

- Reuse `canAttemptOptional(state, config)` from `domain/usage/usageGuard.ts` verbatim.
  No change to the domain module.
- Objective attachment does **not** increment `usageCount`. The visible
  `Generations: N / cap` counter remains one-per-user-prompt.
- Fake provider (`enabled === false`) → `canAttemptOptional` returns `true`; objective
  generation fires free and deterministically.
- At cap → `canAttemptOptional` returns `false` → silent skip → room loads with no
  quest, no error, no notice.
- The memo bounds objective calls to at most one per distinct `roomId` per session,
  so total optional calls are bounded by unique generated rooms explored (not by
  revisit count). The budget is consumed at the rate of new rooms entered, not at the
  rate of prompt submissions.

### Failure and degrade behavior

Any of the following yields `null` from `buildGeneratedObjectiveAttachment`, which is
memoized as `null`:

- `canAttemptOptional` returns `false` (budget exhausted)
- Generator times out (`objective-llm-timeout`)
- Generator network error (`objective-llm-request-failed`)
- Generator empty response (`objective-llm-empty-response`)
- Generator returns `null` (fake generator found no eligible candidate)
- `assembleObjective` parse-fails (invalid JSON)
- `assembleObjective` schema-fails (invalid structure)
- `assembleObjective` satisfiability-drops the condition (hallucinated or ineligible
  `objectId`)
- `assembleObjective` returns `spec: null` for any other reason

In every case: room is visible and fully playable, tracker is hidden, NPC uses default
greeting. **`assembleObjective` is never weakened.** No error notice is shown.

### Logging

Safe log additions only — fixed messages, counts, booleans, structural codes. No room
names, object names, ids, hint text, provider bodies, or API keys.

```
logger.info('optional objective generation allowed', { count, cap, roomId })
logger.info('optional objective generation skipped', { count, cap, roomId, reason: 'usage-cap' })
logger.debug('per-room objective attached', { roomId, attached: boolean })
logger.debug('per-room objective stale', { roomId })   // when stale result discarded
```

`roomId` in this context is the structural navigation id already logged by
`NavigationService`; it is not narrative content.

## Deferred (explicit out-of-scope for v0)

- **`generated-quest-save-load-v0`:** persisting per-room objectives across save/load.
  Generated objectives are session-local only in v0; the tracker does not reappear
  after load. Progress flags that existing interactions already set survive save/load
  (as today); only the tracker display is lost.
- **`generated-story-threading-v0`:** cross-room narrative continuity, objectives that
  reference prior rooms, multi-step quest chains, story state.
- **`generated-mechanical-gates-v0`:** navigation locks, quest-completion-required
  transitions, mechanical consequence of completing a per-room objective.
- Generated `resolve-encounter` / `visit-room` objective kinds in adjacent rooms.
- Per-kind usage caps or a second usage meter for objective calls.
- Real (network-backed) adjacent room generation. Adjacent rooms use `FakeRoomGenerator`
  only in v0.
- Server-side or hosted objective provider.

---

## Consequences

- Prompt-generated plays now feel purposeful across multiple rooms. Each freshly entered
  generated adjacent room can present one small local story beat.
- The `QuestTracker` briefly hides on entering a new room while the async call is
  pending, then appears (or stays hidden if no objective was found). This transient
  blank is the v0 UX trade-off for non-blocking room entry.
- The usage budget is consumed at the rate of unique generated rooms explored, bounded
  by the memo. Users exploring many rooms will exhaust `canAttemptOptional` before the
  at-cap confirm-to-continue gate, at which point rooms load without objectives silently.
- Demo quest, authored bootstrap, save/load, the renderer, the world session, and all
  safety boundaries remain strictly unchanged.
